## Gefundene Bugs — GEFIXT (branch `fix/nlm-extract-robustness`)

### Bug 1 — pydantic/audioop crasht auf Python 3.13
`transcribe` brach beim Import: `ModuleNotFoundError: No module named 'pyaudioop'`.
Python 3.13 hat stdlib `audioop` entfernt (PEP 594); `pydub` braucht es.
**Fix:** `audioop-lts==0.2.2` in den venv (`uv pip install audioop-lts`).
**Follow-up (offen):** sauber als `audioop-lts; python_version>='3.13'` in
`services/data-ingestion/pyproject.toml` eintragen — sonst bricht es bei jedem venv-Resync wieder.

### Bug 2 — extract-Timeout 120s hartcodiert, ignoriert Config, zu knapp
Alle 5 Extraktionen failten mit **leerer** Fehlermeldung = httpx-ReadTimeout.
Ursache: `extract.py` hartcodierte `timeout=120.0` und ignorierte die Config
`ingestion_vllm_timeout`. Der Spark (35B MoE) ist mit der **live RSS-Pipeline geteilt**
→ eine Extraktion dauert gemessen **~160s** → 120s köpfte jede Anfrage.
**Fix (3 Dateien + 2 Tests):**
- `config.py`: `ingestion_vllm_timeout` 120.0 → **600.0**
- `nlm_ingest/extract.py`: `extract_with_qwen(timeout=600.0)`-Param statt hartcodiert
- `nlm_ingest/cli.py`: reicht `settings.ingestion_vllm_timeout` durch
- `tests/test_config.py` + `tests/test_nlm_extract_url.py`: Default + Passthrough-Tests
- `extract_with_qwen` ist NLM-only (RSS-`pipeline.py` hat eigenen Pfad, setzt Timeout schon korrekt client-seitig) → kleiner Blast-Radius.

## Gefundene Issues — GEFIXT (User-Entscheidung 2026-06-19, TDD, branch `fix/nlm-extract-robustness`)

> User-Schnitt: „jetzt fixen = Prompt v3 + strict response_format + lenient item-skip + max_tokens=8000;
> NICHT jetzt = EVENT/Relation-Enums erweitern". Umgesetzt + zwei-stufig reviewt (code-reviewer: kein Blocker).

### Issue A — Out-of-Enum-Typen killen ganzes Notebook (Drone, F127, Hybrid) → GEFIXT
Beobachtet: 35B emittierte `Entity.type='EVENT'` (`{'name':'Dronevation 2026',…}`) bzw. out-of-enum Relations.
`Entity(**e)`/`Relation(**r)` knallte beim 1. bad Item → ganzes NB weg. **Fix (3 Hebel):**
1. **Strict `response_format` (json_schema, strict:true)** — Entity-`type` & Claim-`type`/`polarity` enum-erzwungen (aus den Literals abgeleitet → kein Drift) ⇒ kein `EVENT` mehr an der Quelle. **Relation-`type` bewusst freier String** (nicht enum-erzwungen) → wir beobachten reale out-of-enum-Relations für eine spätere kuratierte Enum-Erweiterung.
2. **Lenient item-skip** (`_build_items`): bad Item wird übersprungen+geloggt, Rest bleibt; per-kind Counter der skipped types (`nlm_extract_skipped_summary`) = das Signal für die Relation-Taxonomie-Runde. Inkl. null-array-Guard (`"entities":null` crasht nicht).
3. **v3-Prompt** (intel-codebook-curator): LOCATION ergänzt (Drift behoben, 13/13 match + Drift-Guard-Test), hartes „nur diese 13"-Enforcement, Events→Claims.
> EVENT NICHT in EntityType aufgenommen (User-Entscheidung). Relation-Enum NICHT erweitert — erst Daten sammeln.

### Issue B — max_tokens=4000 schneidet JSON ab (Pentagon) → GEFIXT
`max_tokens` 4000→**8000**, via `Settings.ingestion_max_tokens` konfigurierbar + an `extract_with_qwen` durchgereicht.
(Restrisiko: falls 8000 bei sehr langen Quellen noch reißt, failt isoliert nur 1 NB — kein finish_reason-Check, Folge-Nicety.)
